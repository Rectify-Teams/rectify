# Rectify vs React — Performance Benchmark

This page embeds the live benchmark suite that measures Rectify and React side-by-side on the same set of operations using identical data-generation logic and an accurate, framework-agnostic timing approach.

## How timing works

Each operation is measured from the moment `setState` is called to immediately after the browser has painted the result. The timer starts at `performance.now()` just before `setState`, and resolves inside `useLayoutEffect` (which fires synchronously after the DOM is committed) followed by one `requestAnimationFrame` to include the paint flush.

```
t0 = performance.now()
setState(updater)         ← schedules work
  ↓ scheduler runs
  ↓ reconcile + commit
  ↓ useLayoutEffect fires (after DOM mutations, before paint)
     ↓ rAF → resolve(performance.now() - t0)
```

This approach is fair to both frameworks because `useLayoutEffect` is guaranteed by both React and Rectify to fire synchronously after every commit.

## Operations

| Operation | Description |
|-----------|-------------|
| Create 1,000 rows | Mount 1,000 new rows from scratch |
| Create 10,000 rows | Mount 10,000 new rows from scratch |
| Replace all rows | Swap the entire row list with a new set of 1,000 rows |
| Update every 10th row | Append `" !!!"` to the label of every 10th row |
| Select a row | Mark row #5 as selected (changes its highlight) |
| Swap rows | Swap row #2 and the last row |
| Append 1,000 rows | Add 1,000 rows to the existing list |
| Remove a row | Delete row #5 from the list |
| Clear all rows | Remove every row |

## Why Rectify can be faster

Rectify's architecture is intentionally minimal compared to React 18's full concurrent runtime:

- **Single-pass commit** — Rectify walks the fiber tree once. React runs three separate traversals (before-mutation, mutation, layout-effects).
- **Flat effect arrays** — Layout effects are pushed into a plain array during render and iterated after commit. React re-walks the fiber tree to collect `useLayoutEffect` hooks from each fiber's circular `updateQueue`.
- **Direct microtask scheduling** — `SyncLane` updates go through `queueMicrotask()`. React routes the same updates through its own scheduler's min-heap priority queue before posting a MessageChannel task.
- **No concurrent mode overhead** — Rectify has no offscreen trees, no hydration paths, and no DevTools hook callbacks, so every commit executes less code.

React carries all of this infrastructure even when unused, which means results in a micro-benchmark like this overstate the real-world difference.
